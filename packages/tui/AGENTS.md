# Agent instructions

`stet` is a read-only companion TUI for CLI coding agents (claude code, opencode, codex). You run an agent in one terminal pane and `stet` in another to inspect what changed: everything an IDE shows you, nothing it does for you. **It only inspects. It never reviews, explains, approves, rejects, gates, talks to an agent, or writes to the repo.** That constraint shapes every change here.

See `README.md` for what stet does, its keys, and its non-goals; see `SPEC.md` for the implementation invariants behind those features; see `TASTE.md` for the craft bar every interface surface clears. Read SPEC before changing tree construction, the viewer, recency, diagnostics, scopes, or worktrees, and TASTE before adding or reworking any UI surface.

## Stack

- Bun for runtime, scripts, dependencies, tests, and build.
- TypeScript with `strict` enabled.
- `@opentui/core` and `@opentui/solid` (SolidJS fine-grained reactivity); JSX configured with `jsxImportSource: "@opentui/solid"`, and `@opentui/solid/preload` in `bunfig.toml`. Solid is the rendering layer OpenTUI's own authors use; its surgical updates avoid the whole-tree re-render churn that a React reconciler submits to the native renderer.
- Git output is the source of truth for the file map: the app shell paints immediately from the empty model and the initial git load seeds it in the background (so a large repo shows the UI at once instead of a blank screen during the load), and a filesystem watcher (debounced, over the worktree tree plus the resolved git dir) keeps it current, with an adaptive safety poll as the floor that covers anything the watcher misses. Refresh triggers feed conflating sliding-1 queues, so a burst of changes collapses to one in-flight read plus one trailing run rather than a backlog; the changed set drains on every tick, while the full repo-file list re-lists only when the changed path-set shifts (plus a slow floor), not on a blind timer. The git-backed file map renders before any checker or diagnostic resolves; diagnostics are independent async decorations over the stable git file list, never blocking it.
- The diff/file viewer is stet-owned (`src/diff/`, `src/components/diff/DiffView.tsx`): `@pierre/diffs` is the headless engine (patch parse + Shiki syntax highlight, WASM oniguruma), and stet hand-renders the windowed rows on OpenTUI primitives rather than using the built-in `<diff>`. Syntax colors come from stet's own Shiki theme (`src/theme/shiki.ts`); there is no tree-sitter path. The patch string the engine parses is produced in-process by jsdiff (`diff`) from the scope's two endpoints (`src/git/file-patch.ts`), not by `git diff <ref> -- <path>`: git's pathspec-limited diff-index costs seconds per open in very large repos (#188) while blob/worktree reads stay O(path depth). The pathspec `git diff` survives only as the fallback in `Git.fileDiff` for sides an in-process diff can't reproduce (missing/oversized, jsdiff timeout, one-sided CRLF, zero-hunks-vs-numstat). See SPEC's Viewer section.
- Clipboard copy uses `pbcopy` on macOS and a Wayland/X11 tool (`wl-copy`, `xclip`, or `xsel`) on Linux.

## Conventions

- Read the relevant `SKILL.md` under `.agents/skills` before related work: `bun` before any dependency/script/build change, `opentui` before any OpenTUI code.
- Prefer `bun run`, `bun test`, `bun install`, `bun add`, `bun add -d`, `bun remove`, `bun build`. No Node/npm/Jest/esbuild wrappers unless explicitly requested. Runtime flags go before `run` (e.g. `bun --watch run <script>`).
- Keep `bun.lock` changes paired with dependency changes. Declare direct dependencies in `package.json`; don't rely on transitive ones.
- Pin every `dependencies` and `devDependencies` entry to an exact version (no `^`/`~`). stet ships as a compiled binary (`private: true`, `bun build --compile`), so the manifest mirrors the lock and every bump is an explicit, reviewable edit; caret ranges only help library consumers dedupe, which never applies here.
- Lint and format are oxlint and oxfmt, run from the repo root (`bun run lint`, `bun run fmt`), not from within `packages/tui`: both tools resolve this package's `.oxlintrc.json`/`.oxfmtrc.json` on their own when invoked from the root, so a single root-level command already covers the whole repo. Do not add ESLint or Prettier.
- Dead exports are caught by knip (`bun run knip`, also part of `bun run check`). Remove the `export` keyword rather than suppressing.
- Git hooks run through lefthook (`lefthook.yml`, wired up automatically by its own postinstall on every `bun install`, no extra step). `pre-commit` runs on staged files only: both format and lint auto-fix and re-stage (oxlint's `--fix` only applies safe, unambiguous fixes), and anything neither can safely fix still blocks the commit. `pre-push` runs only `typecheck` (root `bun run typecheck`, fanning out to both `packages/tui` and `docs` via `bun run --filter '*' typecheck`) and `knip` (tui only, per `knip.json`'s `ignoreWorkspaces`), in parallel: those are the two checks that need the whole project graph and can't be scoped to staged files. `bun test` and full-project lint/format are deliberately left off pre-push (pre-commit already covers staged-file lint/format, and CI runs the full `bun run check` on every push/PR), since re-running the whole suite on every push was slow enough to be worth trading for CI as the actual safety net. Agents run the same hooks as humans; no skip/bypass config is layered on top of git's own `--no-verify` or lefthook's `LEFTHOOK=0`.
- Use `===` and `!==`; never `== null` or `!= null`.
- No explicit return type annotations unless needed for exported API clarity, recursion, overloads, or inference limits.
- Rely on type inference; avoid explicit annotations or interfaces unless needed for exports or clarity, and inline single-use values rather than naming them.
- Prefer `const`; use ternaries or early returns instead of reassignment, and early returns instead of `else`. Don't introduce a `let` just to hold a value for a later statement (e.g. across a `try`/`catch`); lift it into a helper or expression (wrap a throwing call in a `Result` rather than a mutable placeholder).
- Prefer functional array methods (`map`/`filter`/`flatMap`) over loops; use type guards on `filter` to keep inference downstream.
- For generic data utilities, prefer native ES2024 over a third-party util library: `Map.groupBy`/`Object.groupBy` for grouping, in preference to a hand-rolled get-or-create-push loop (`Map.groupBy` returns a `Map`, which a `Record`-returning lib helper does not), `new Set`/`Set` methods (`union`/`difference`/`intersection`) for dedupe and set algebra, `toSorted` with an explicit comparator (use `localeCompare` for locale-aware order), and `structuredClone` for deep copies. Project-specific pure helpers (no Effect/Solid/OpenTUI imports) live under `src/utils/`; domain helpers stay in their domain module.
- Use Bun APIs where they exist (`Bun.file`, and so on).
- Never alias imports (no `import { x as y }`).
- Use the `@/` path alias (mapped to `src/` in `tsconfig.json`) for cross-directory imports (`@/state`, `@/diff/follow`), not upward-relative chains (`../../diff/follow`). Keep same-directory `./sibling` imports relative, since they signal colocation. Tests import source through the alias too (`@/git/activity`), not `../src/...`.
- Prefer dynamic imports for heavy modules on startup-sensitive paths; bind them at the top of the narrowest scope that needs them, and keep branch-specific imports inside their branch so the git tree renders before anything heavy loads.
- Do not reach for `as`, `!`, or `any` without first exhausting proper solutions.
- Do not silence lint or type errors with rule overrides or casts. Fix the root cause.
- Comments are JSDoc or `TODO`/`FIXME` only; keep them sparse and useful, and do not narrate obvious code. JSDoc is not a license to narrate: a `/** ... */` that restates the signature or types, or says what the code plainly does, is still noise. Comment the why, not the what.
- Prefer small typed modules (git parsing, diagnostics, clipboard, CLI args, UI state), and structured parsing over ad hoc string manipulation when a command offers machine-readable output.
- Visual colors come from theme tokens in `src/theme` (`tokens.ts` shapes the `Theme` and is its `Schema` single source of truth, `dark.ts`/`light.ts` hold the hexes, `resolve.ts` exposes them via `useTheme`), never hardcoded hex at call sites. Each `<scrollbox>` must set `scrollbarOptions`: the thumb comes from the `scrollbar` token and the track stays `theme.rgba.transparent` so it inherits whatever surface it scrolls over (one track color can't match every surface). Without `scrollbarOptions` OpenTUI paints a loud default gray that ignores the palette.
- User-facing messages carry a severity through one taxonomy in `src/log/` (`levels.ts`: the `LogLevel` union plus `levelColor`, mapping to `severity.*`/`success`/`text.secondary` tokens, and `levelGlyph`, the bare text-presentation glyphs `✖ ⚠ ✓ ℹ` shared with the counts badge and problems panel). Severity is always paired with its glyph, never signalled by color alone, so it reads under `NO_COLOR` and for colorblind users. Two renderers consume it: `StatusBar.tsx` colors the status line (`statusRightLevel()` feeds the glyph + `levelColor`), and `log/terminal.ts` (`logError`/`logSuccess`/`logInfo`) prints pre/post-TUI lines, deriving ANSI from the active theme hex via `Bun.color(hex, "ansi")` gated on `Bun.enableANSIColors` (which honors TTY/`NO_COLOR`/`FORCE_COLOR`); `info` stays glyph-only there since the terminal background is uncontrolled. Glyphs must stay bare codepoints (no `U+FE0F`), or the variation-selector form renders two rows tall.
- The status bar's right slot (`StatusBar.tsx`) reads as two parallel groups, each led by its own mark: the changed file (a `RecencyDot` leading its path) and the outcome (the severity glyph leading its message), divided by a plain gap (space groups better than a middot). The most recent changed file is presented with the tree's changed-file cue rather than a neutral path or a number: the path is tinted by its git change kind (`gitModel().changedByPath.get(path)?.kind` -> `theme.colors.kind`) with a leading `RecencyDot`, both fading toward the background (path -> `text.faint`, dot `recency.fresh` -> `recency.aged`) across the 30s `RECENT_MS` window, so freshness reads as brightness and the group recedes silently as it ages (there is no "Ns ago" label). The dot is a shape, so "changed" survives `NO_COLOR` where the color-only kind tint would not; the leveled status still carries the glyph + `levelColor`. `RecencyDot` lives in its own `src/components/RecencyDot.tsx` and renders only the mark (`●`); spacing is the caller's, passed as a `marginLeft`/`marginRight` so it stays gated to the dot's presence (the tree and file picker trail it with `marginLeft`, the status bar leads with `marginRight`). The model (`statusRightModel`) exposes the pieces (`statusRightPath`/`statusRightMessage`/`statusRightRecencyAt`/`statusRightChangeKind`) already width-budgeted against the glyph/dot/gap the bar draws, and adjacent same-color spans share one `<text>` because an empty `<text>` in a flex row paints a phantom cell.
- The three persistent panes (tree, viewer, problems) share one frame, `src/components/PaneFrame.tsx`: a `position:relative` wrapper around a neutral `single` box bordered in `border.unfocused` in every state, with the pane's content as children. Focus is marked by tinting the box's four corner glyphs `border.focused` (pink), drawn as four absolute single-glyph `<text>` overlays gated on `focused` so they overpaint only the corner cells and reserve no space (no reflow on focus). The box border itself never turns pink; a full-pink focused outline was the old, noisy behavior and should not return. A corner glyph is a self-contained cell, which is why corner ticks work where a spine on the border edge cannot: a cell holds one glyph in one color, so a pink vertical `│` on the left edge either pokes past or wraps the neutral top/bottom line, while a pink `┌`/`┐`/`└`/`┘` is a clean joint. This is still a color-only focus signal (the neutral box already draws corners on every pane), the same as before, not a `NO_COLOR` distinction. Overlays (dialogs, comboboxes, menus, CaretCard) keep a full `border.focused` box, since a modal layer is always the active surface.
- Every focused `<input>` must set `focusedTextColor` (and `focusedBackgroundColor`) alongside `textColor`/`backgroundColor`. OpenTUI's input caches separate focused/unfocused colors at construction, and the OpenTUI/Solid reconciler builds the renderable with only `{ id }` then applies props through setters; the `textColor` setter updates only the _unfocused_ color, so a focused input that omits `focusedTextColor` renders its text in the default white regardless of `textColor`. Dark themes mask this (white-on-dark reads fine); light themes show white-on-light typed text. Same applies to any focused-only color: set the `focused*` variant explicitly, never assume it inherits from the base.
- User config lives in `src/config/` (a normal domain: pure `paths`/`schema`/`load` plus a `Config` service), read from `~/.config/stet/config.jsonc` (or `config.json`; `configPaths` lists candidates in priority order, both parsed as JSONC) once at startup. It loads on its own short-lived `ManagedRuntime.make(ConfigLive)` in `main.tsx` _before_ the app runtime's first use, because the active theme must be set before the diff highlighter warm-up reads it; a missing/invalid config falls back to defaults plus a notice and never blocks startup. Named themes resolve through `src/theme/registry.ts` (`resolveThemes` validates each entry, applying a `{ base, ...overrides }` merge with cycle detection, and resolves its `syntax` (a bundled Shiki theme name, per-token overrides, or inherited from the base); `themeForName`/`syntaxThemeForName` are the lookups with a dark fallback) and `selectThemeName` picks the active one from the config selection and the appearance. Theme state is reactive (`src/theme/active.ts`, a `createRoot` of signals): `appearance` follows the terminal (seeded at startup, updated live by the renderer's `theme_mode` event in `App.tsx`), `selection` is the config choice, and `activeTheme`/`activeThemeName` are memos over both. `useTheme` returns a stable handle whose `colors`/`rgba` getters read the `activeTheme` memo, so the existing call sites (which read `theme.colors.x` in JSX/thunks) re-theme on a runtime appearance flip with no change. The diff engine reads `active.ts` too (not `state`): it attaches the active render theme on demand (a per-name custom theme from tokens, or the bundled theme named by `syntax`) and keys its highlight cache by it, and the diff render effect in `state.ts` re-runs on `activeThemeName`, so a flip re-renders the diff without stale colors. This is the #101 follow-up, now landed. The `t` theme switcher (`src/components/ThemeCombobox.tsx`, mirroring the `FileCombobox` overlay) writes the same `selection` signal live: `themeComboboxResults` (a filtered `themeComboboxItems` over `themeNames()`, with an `auto` row for the `undefined` selection) drives a preview effect in `state.ts` that re-themes the whole UI while the picker is open, fed by keys, hover (`onMouseOver`), and the wheel through one `themeComboboxIndex`; `closeThemePicker(commit)` either keeps the highlighted selection or restores the `themeComboboxOrigin` captured on open. It is session-only (no config write); `themeNames()` and the `selection` accessor are the only new `registry.ts`/`active.ts` exports it needs. Configured `languages` follow the same startup path: `resolveLanguages` in `src/diagnostics/servers.ts` (the theme-registry pattern: per-field merge over built-ins, issues surfaced as notices) feeds `registerServers`/`registerLanguages` in `main.tsx` right after themes; inline servers are repo-local/PATH-only, never provisioned, and the trust boundary is the user's global config, never the inspected repo.
- The tree's file-type icons resolve through `src/utils/file-icon.ts` (`fileIcon`/`folderIcon`), a Nerd Font lookup keyed exact-filename then extension, then a config glyph for any unmatched leading-dot dotfile, then a default glyph. They are on by default, gated by `state.iconsEnabled()` and the `--no-icons` flag, and rendered monochrome (`text.muted`) for now; per-type icon colors via a `theme.colors.icon` token group are a deliberate follow-up. Every file-leaf icon renders through the shared `src/components/FileIcon.tsx` (the fixed 2-cell box, self-gating on `iconsEnabled`, handling the symlink glyph): the tree's `FileRow`, the file picker, the search headers, the references headers, and the problems headers all use it, so a file reads with its icon on every list surface. Directory/collapse glyphs stay inline in `TreeRow` (not file leaves).
- The header bar (`src/components/HeaderBar.tsx`) is scope-aware git identity on the left, view stats on the right, budgeted to `terminalWidth()` so it never clips. Left is context: in the main checkout `repo · branch` (repo = the stable `mainWorktreePath` basename), in a linked worktree `worktree-folder · branch` (the folder is the primary token, dropping the branch when it strictly equals the folder to avoid duplication), each with a leading Material Design Nerd Font glyph (`md-source_repository` U+F0CCF vs `md-file_tree` U+F0645) and an `md-source_branch` (U+F062C) glyph on the branch. Glyphs are decorative (icon-gated on `iconsEnabled`, dropped under `--no-icons` where the text carries the meaning), code points verbatim from glyphnames.json. A `commit` scope drops the identity entirely (a pinned commit is unrelated to HEAD's branch) and shows the full commit subject in the left slot spanning the width up to the right stats, truncating with an ellipsis only when it does not fit; the branch (non-commit) and subject (commit) are the flexible tokens the width budget shrinks first, never the changed count. The current branch rides live on `GitModel.branch`, parsed from the `##` header of `git status --porcelain=v1 -b -z` (`parseBranch` in `src/git/model.ts`), so it refreshes on the same drain as the changed set.
- A binary file renders a metadata surface, not raw bytes and not a blank pane: `src/components/BinaryPreview.tsx` centers the file type, image dimensions, and human-readable size (an old -> new delta in diff mode), the metadata an IDE shows for an image stet cannot draw as pixels (OpenTUI has no SIXEL/Kitty path, anomalyco/opentui#92). Dimensions come from `src/file/image-meta.ts` (`imageMeta`, a pure magic-byte header read for PNG/JPEG/GIF/WebP/BMP, no decode; SVG stays text), threaded through the enriched `binary` variants of `FileContent` (`src/file/content.ts`) and `SideContent` (`src/git/file-patch.ts`). Diff mode has no patch, so `git.binaryMeta` fetches both sides' `SideMeta` (reusing `fileDiffSides`/`fetchSide`) onto the `DiffView` snapshot's `binary` field; the `Viewer` gate prefers `BinaryPreview` over `DiffView` whenever a side is binary, in either view mode.
- Any surface showing a source line highlights it through the diff's shared `highlightSnippet` (`src/diff/engine.ts`), never plain text: the diff, the search results, the hover card, and the references previews. The shared renderer is `src/components/CodeLine.tsx` (one `StyledText` buffer per line, windowed to a width), fed by a per-surface span cache that invalidates on new content or `activeThemeName()` and upgrades rows plain -> highlighted in place (identical text, no layout shift).
- Keybindings live in one registry, `src/help/keys.ts` (`KEY_HELP`, ordered `{ heading, entries }` groups). Both the in-app `?` overlay (`HelpDialog`) and `stet --help` (`helpText()` -> `keyBindingsHelp()` in `src/cli.ts`) render from it, so the two can't drift; add or edit a key there, never in either renderer.
- Conventional commit style for commits and PR titles: `type(scope): summary`, with types `feat`/`fix`/`docs`/`chore`/`refactor`/`test` and an optional scope. Commits go through `bunx gitzy` (see "Implementing a change").
- No AI-generated sign-offs in commits, PR text, docs, or generated content.

## State and Effect conventions

Reactive state is SolidJS-native (signals/memos); async IO is Effect v4 (beta). `effect` is pinned to an exact beta version; `effect/unstable/*` can break on minor bumps. The two layers meet only at one seam: the long-lived `ManagedRuntime` in `src/runtime.ts`.

- Wrap existing pure functions in services; do not rewrite the pure logic, so its tests stay intact.
- Define services with `Context.Service` plus a `Layer` (there is no `Effect.Service` in v4); define errors with `Data.TaggedError`, one tag per distinct failure. Code is organized by domain folder under `src/` (`git/`, `file/`, `diagnostics/`, `diff/`, `clipboard/`, `watcher/`): each colocates its pure modules with the Effect service that wraps them in a `service.ts`. The service `Layer`s are still assembled once into `ManagedRuntime.make(AppLayer)` in `src/runtime.ts` (the single assembly point); run service effects through `runtime.runPromise`/`runtime.runFork`, never with a fresh `Effect.runPromise` + `Layer.provide` per call.
- Prefer the data-last pipe form for combinators (`x.pipe(Effect.flatMap(f))`) over the data-first form (`Effect.flatMap(x, f)`); the two-argument data-first form trips `unicorn/no-array-method-this-argument`.
- All app state lives in `src/state.ts`: one global `createRoot` exposing `createSignal` primitives, `createMemo` derivations, and `createEffect`-driven async flows, exported as the `state` object. Components and the keymap import `state` and read accessors directly (`state.selectedPath()`), so reactivity is fine-grained: only the renderables whose inputs changed update. `App` owns the render-tree/renderer concerns `state` deliberately does not: keyboard wiring, terminal-dimension sync, the terminal title, and the deleted-worktree recovery effect (it calls `renderer.destroy()`). `main.tsx` seeds the signals (in one `batch`) before `render`.
- Async data flows are Solid effects that run an Effect through the runtime and write a signal: the diff/content pipeline is the load-critical one. The diff is rendered by stet's own windowed renderer (`src/diff/`, `src/components/diff/DiffView.tsx`) over `@pierre/diffs` (headless parse + Shiki highlight), not OpenTUI's built-in `<diff>` (whose gutter-width oscillation wedged the scheduler). `createEffect` keyed on the selection forks the load via `runtime.runPromise(effect, { signal })` and commits the `diffView` snapshot in two structure-identical phases: first plain rows (parse only, instant), then a rows upgrade once the async Shiki highlight resolves (path- and abort-guarded). The previous snapshot is held until phase 1 resolves, so the renderer never receives empty/stale/partial content, and both phases share the same row count and fixed gutter width so the swap never thrashes layout. Keep diff updates coherent and one-per-selection.
- Cancellation is via the `AbortController` passed to `runtime.runPromise`'s `signal` and `onCleanup` (Solid re-runs the effect on dependency change, aborting the prior — which interrupts the Process fiber and kills any in-flight git). Background fibers (the watcher and safety poll, the quiet re-run) follow the same restart-on-rekey pattern; aborting the controller closes the `fs.watch` handles via the watcher stream's finalizer.
- **Action placement.** A model/data action lives in `state` and is reached directly by components, the keymap, mouse handlers, and effects, never prop-drilled (placing one in `App` and threading it down was the `switchWorktree` bug). An action with an irreversible host side-effect (the `renderer` or `process`) is injected into the keymap through its `HostEffects` seam, and reached inside components via `useRenderer()` (as `DiffView` does for scroll/measure). Never put the `renderer` in `state`: it is a render-tree resource, and keeping it out is what keeps `state` testable and the keymap's `quit` injectable.
- The keymap is `createKeyHandler(host)` (`host: HostEffects` carries only `quit`); it reads/writes `state` directly and wraps each dispatch in Solid `batch` so a keypress is one update. Mouse is wired symmetrically: row components carry `onMouseDown` handlers (closures over their own row) that dispatch into the same `state` signals, each in `batch`, so a click mirrors the equivalent key. Every action except `quit` lives in `state` (including `switchWorktree`, so the keymap, the worktree picker's click, and App's recovery effect all reach it directly). `quit` calls `renderer.destroy()` then `process.exit(0)` (logging any message after `destroy()`, so it lands on the restored screen, not the alt buffer); the background poll keeps the event loop alive otherwise. Every quit trigger routes through this one `quit`: `q`/`esc` and ctrl-c alike, and the keyboard path additionally prints the post-exit "new release available" notice, but only when the background check found a newer release (the worktree-recovery exit calls `quit` without it, and the OS-signal/crash backstops never route here, so both stay silent). `q` does not quit directly: it opens the `quitConfirmOpen` alert dialog (`QuitConfirm.tsx`, an early keymap guard) that only calls `quit` on `y`/`enter`, so no single keystroke ends the app; `esc` never quits (it only ever closes: the problems panel, then the search view, then nothing), and ctrl-c stays instant (checked before the confirm guard) as the deliberate escape hatch. `QuitConfirm` is stet's one true **modal**, distinct from the dismissible picker/palette overlays (help, comboboxes, scope/command menus, references/symbols): an alert dialog is a decision surface, so it renders a full-screen barrier that dims the app with `surface.scrim` (an opaque theme token blended to a fixed alpha in `resolve.ts` as `rgba.scrim`) and captures background mouse-down, keeping the app inert and never dismissing on an outside click (the alert-vs-dialog line). The pickers stay casually dismissible and add no barrier, because they are browsing surfaces, not decisions. `render` sets `exitOnCtrlC: false` and the keymap handles ctrl-c itself, because OpenTUI's built-in `exitOnCtrlC` only calls `destroy()` (never `process.exit`), so it hangs behind the poll instead of exiting. An OS-signal backstop in `main.tsx` (`SIGINT`/`SIGTERM`/`SIGHUP`, plus `uncaughtException`/`unhandledRejection`) restores the terminal and exits even when a wedged input loop never delivers a keyboard quit, so an external kill or a crash can't leave the alt-screen buffer active and the next launch blank; `renderer.destroy()` is idempotent, so the backstop composes with the keymap's own `quit`.
- **The main area is a view mode, not a stack of booleans.** `mainView` in `state` is a union (`"file" | "search"`) rendered by a `Switch` inside the `Viewer`'s border, so exactly one view ever owns the pane interior; a future pane extends the union, adds a `Match`, and gets a keymap branch, never another `xOpen` flag with implicit precedence. View-local state (the search pane's query/results/selection) lives in its own signals and simply persists when `mainView` returns to `"file"`, which is how close-and-reopen restores the pane for free. `focusedPane` carries a matching value (`"search"`) so the keymap's modal chain and the Viewer border react to it like any pane.
- **Viewer navigation is tabs over a pure history model.** `src/viewer/navigation.ts` is a pure module (no Solid/Effect/OpenTUI) holding `NavState` (tabs, each an ordered history of `Location`s, plus a per-path MRU viewport); it unit-tests like `git/tree`. `state` keeps the live signals (`selectedPath`/`cursorIndex`/`cursorColumn`/`viewerScrollTop`/`viewerScrollX`) as the rendering source of truth and uses `navState` as an orthogonal **recorder**: a navigation captures the leaving `Location` into the active tab + MRU (`captureCurrent`/`recordCurrent`/`remember`), then applies the arriving one by setting the live signals plus a single `pendingRestore` that the `Viewer` effect consumes once the diff has loaded under the matching view intent. That one path replaced the old scattered resets, so "reset on file switch" is just restore-to-default. **Invariant:** at most one `preview` tab; all navigation routes to it (browse coalesces in place, a jump pushes), `ctrl-t` toggles pin/unpin, and navigating to a file already in a pinned tab focuses that tab (never duplicates). `selectFile`/`moveFocus` and the tab actions (`togglePinActiveTab`/`closeActiveTab`/`cycleTab`/`activateTab`) all live in `state`, reached by keymap/mouse/effects like every other action; cursor is stored as a file line (stable across refreshes), scroll is best-effort. The in-line caret (`cursorColumn`) joins those captured-on-leave signals: it is a word-granular caret stored as a precise UTF-16 offset (`src/diff/words.ts` is the pure word model), re-homed to the cursor line's first word by `setCursorRow` on every vertical move and clamped on content reload, so a diagnostic jump still lands on its exact column.
- **The viewer's line selection is a transient anchor over the caret.** `selectionAnchor` (a navIndex, or `undefined`) is a sibling to `caretLineLevel`: the selection spans `[min, max]` of anchor..`cursorIndex`, whole-line, and `selectionRange`/`selectionText` derive it. It is built by `extendSelectionTo` (Shift+↑/↓ in the keymap, and a shift-click or drag in `DiffView`) and cleared by every plain caret move (`setCursorRow`, the `caretNextWord`/`caretPrevWord` word hops) and every navigation (`goToLocation`, the one choke point), plus `esc` ahead of find/global. It is never captured into navigation history. `copySelection` (the `C` key) copies the spanned lines' `content` (no sign, no gutter), falling back to the caret line when there is no selection; a folded/git-elided region is excluded for free since it never enters `navigableLines`. The band reuses the cursor lift in `DiffView`'s `resolveBackground` (selected rows pass `isCursor(row) || isSelected(row)`), so it is theme-correct per diff state with no new token, and the focus row still shows its caret word on top. The line row's **text leaves** (the gutter `<text>`, the content `<box>`, and the `StyledLine` `<text>`, plus the marker `<text>`) set `selectable = false` via `ref`, not just the row box: OpenTUI starts its own native text selection on whatever renderable a left-drag hit-tests, gated on _that element's_ `selectable`, so marking only the box leaves the text leaf selectable and a drag paints OpenTUI's colorful per-token highlight over our band. Drag-to-select is ours, not OpenTUI's: once native selection is off, OpenTUI still delivers `drag` events but pins them to the first-captured row (its `line()` can't follow the pointer), so `DiffView` captures a `dragOrigin` (`rows()` index + press `y`) on the plain `onMouseDown` and, on each `onMouseDrag`, derives the target row from the live `event.y` (`navIndexForRow` snaps to the nearest line row so folds/gaps are skipped), precise for height-1 rows and approximate in wrap mode like the existing wrap caret-click. Drag, not shift-click, is the portable mouse range gesture: many terminals (macOS Terminal.app, iTerm2, xterm) reserve **Shift+mouse for their own text selection and never forward it**, so a shift-click extend is best-effort (it works where the terminal forwards the modifier, and degrades to a plain caret-move click where it doesn't); drag and Shift+↑↓ are the reliable ones. Actions live in `state`, reached by keymap/mouse like every other.
- **Folding and gap expansion are one "collapsed regions" seam.** `src/diff/regions.ts` is a pure module (no Solid/Effect/OpenTUI, unit-tested like `git/tree`): `foldRegionsFor` dispatches the region model on the file's language (`computeFoldRegions` indent-based for code, `computeMarkdownFoldRegions` heading-based for markdown; `state` derives the mode from `languageForPath`), and `applyCollapsedRegions` rewrites the raw row stream for the current collapsed state, emitting a non-navigable `marker` row (added to the `DiffRow` union) for each folded block or git-elided gap, revealing a gap's lines from a loaded source, and **re-densifying each surviving line row's `navIndex` while rebuilding `navigable`**. That re-densification is the load-bearing invariant: the caret (`cursorIndex`) indexes only visible lines, so it can never land inside a fold. `state` holds two `Set`s with opposite defaults (`foldedRegions` present = collapsed, `expandedGaps` present = expanded) plus a lazily-loaded `gapSource`; `collapsedRender` is the memo that feeds both `navigableLines` and the viewer's rows, keyed on the fold/gap sets and source (not the caret, so cursor moves don't recompute it). `truncated`/`truncatedHidden` stay on the raw `render` (the maxLines cap is about the whole file, not what the user folded). Toggle actions (`toggleFold`/`toggleGap`/`toggleRegionAtCaret`, reached by keymap/mouse like every other action) remap the caret against the next navigable computed purely in the same batch (`remapCursorAfterToggle`), and `revealLineForJump` clears a fold covering a jump target so jumps still land. A landed jump also consumes `pendingRestore` so a later restore pass can't overwrite the caret (the two effects' ordering is not guaranteed once folds churn `navigableLines`). Gap source is retrieved by the git service's `fileSource` (reusing `fileDiffSides`/`fetchSide`), lazy on first expansion, never on the diff hot path. Keys: `z` folds/unfolds/expands at the caret (wrap moved to `x`).
- **Caret-anchored decorations go through one content-agnostic seam, distinct from the centered overlays.** The hover card (`K`) is the first consumer; peek/gutters reuse the same path. The placement math is pure (`src/viewer/anchor.ts`: `caretCell` maps the caret to its on-screen cell or `undefined` when scrolled out, `placeCard` clamps the card to the viewer and flips it above near the bottom edge), unit-tested like `src/diff/follow.ts`. The card (`src/components/diff/CaretCard.tsx`) mounts **inside `DiffView`** (a relative box wrapping the scrollbox) so it positions in viewer-content coordinates with no dependency on header/status heights and never clips against the scrollbox. One `viewerDecoration` signal in `state` carries a status plus `DecorationLine[]` (so it stays consumer-agnostic, named `viewerDecoration` not `hover`): each line is a highlighted **code** line (`RenderSpan[]`, rendered as a `StyledText` like the diff's `StyledLine`) or a plain **prose** line, so a consumer can mix syntax-highlighted snippets with text. Hover highlights its code blocks through `highlightSnippet` (`src/diff/engine.ts`), which reuses the diff's shared Shiki highlighter and active theme, so the card and diff share one palette. `openViewerDecoration` captures the caret/scroll/file/scope/repo/theme anchor, `resolveViewerDecoration` updates content only while that anchor still holds (a no-op once a caret move closed it, so a late async result can't resurrect the card), and an orthogonal clear effect closes it on any caret/scroll/file/scope/worktree/theme drift. `esc` closes it ahead of the find and global handlers. The card designs every status (loading/ready/empty/error) and reserves no document space, so it never shifts the diff. New `state` actions are reached by the keymap/effects like every other action, never prop-drilled.
- **The context menu is a thin dispatcher, shared across two panes.** Right-click (a tree row or a viewer symbol) or `Shift+F10` opens a menu whose items only dispatch existing self-guarding `state` actions; it introduces no new behavior. The item list is a pure model (`src/components/command-menu/items.ts`, `buildCommandMenuItems`, unit-tested like `viewer/anchor.ts`) that `state.commandMenuItems` wraps over the live caret/tree signals, so the render and the dispatch (keymap `return` and row click) read the same list and the highlighted index always maps to one action. One presentational `src/components/CommandMenu.tsx` renders it, placed by `placeCard` (`src/viewer/anchor.ts`) in two coordinate frames: the viewer instance mounts **inside `DiffView`** and anchors to the caret via `caretCell` (exactly as `CaretCard` does), the tree instance mounts in `App` and anchors in global terminal coordinates at the clicked/`focusedRow` cell. Shared `commandMenuOpen`/`commandMenuIndex`/`commandMenuContext`/`commandMenuAnchor` signals (only one instance is ever open) gate a keymap modal block placed right after the scope-menu block, so it owns the keyboard while open. Right-click is detected by `event.button` through `isRightClick` (`src/components/mouse.ts`, the one place that reconciles OpenTUI's `number` button with `MouseButton.RIGHT`); a right-press focuses the node/caret then opens the menu, a left-press keeps today's behavior. Every item dispatches through `state.dispatchCommandAction` except `openEditor`, which needs the renderer: `src/editor/open.ts` holds the shared `openInEditor`, reached by the keymap's `HostEffects` (App wraps it with its renderer) and by the menu's row click (via `useRenderer()`), so the renderer is never prop-drilled. Opening captures a `commandMenuGuard` snapshot (pane, file, focused node, caret, scroll, scope, repo) and a clear effect `closeCommandMenu()`s the instant any of it drifts, exactly like the hover card's `decorationAnchor`: this is the invariant that keeps "menu open" from outliving the anchored render (a click elsewhere moves the caret/focus and dismisses the menu, so it can never linger invisibly and swallow keys). Inapplicable actions are omitted from the list rather than shown disabled (intel only when the caret is on a symbol; Show full content only when the file is truncated; a directory offers just Copy path), so every visible row acts; the highlighted row carries a `▸` caret so the selection reads under `NO_COLOR`.
- **stet owns focus and selection, so disable OpenTUI's.** The renderer is created with `autoFocus: false` (`src/main.tsx`) so a click never native-focuses a renderable, which would draw the terminal cursor in it; an `App` effect hides the cursor whenever no overlay input is open (overlay inputs show their own caret via the explicit `focused` prop). Chrome that is clicked but is not an input (the tab cells, the sidebar tree) sets `selectable`/`focusable` to `false` via a `ref`, so a click or double-click activates it without starting a text selection (a stray highlight) or capturing keys.
- **Long lists window to the viewport; never mount one renderable per row.** The sidebar tree, the search results, the problems panel, and the references overlay render only their visible slice (`sidebarScrollTop`/`searchScrollTop`/`problemsScrollTop`/`referencesScrollTop` plus the viewport height) through `<Index>` in a plain box, with uniform height-1 rows (the problems panel's inter-group blank line is an explicit `spacer` item, not header padding, so the slice math stays uniform). OpenTUI materializes native buffers per mounted element and `viewportCulling` only skips paint, so an unwindowed list rebuilt each refresh tick (fresh row objects through a reference-keyed `<For>`) stalls per tick and eventually exhausts native buffers on large repos. Cursor-follow lives in `state` (`followScrollTop`, margin 2), tracking only the cursor and viewport and reading rows/offset untracked, so a background refresh never snaps a wheel-scrolled window back to the cursor; a clamp effect bounds the window when the row list shrinks. Windowed row components read everything through props (an `<Index>` slot swaps its row in place, and can flip type) and set no renderable `id` (a per-slot changing id confuses the reconciler's anchoring). Per-row derived data must be O(1) lookups into precomputed per-directory maps (`directoryRecencyByPath`, `directorySummariesByPath`), never per-row scans over a whole map. The shared glue is extracted: a new windowed surface reuses `windowWheelHandler` (`src/components/wheel.ts`) for its wheel handler, `followListWindow` (local to `state.ts`) for the follow/clamp effect pair, and `ListScrollbar` (`src/components/ListScrollbar.tsx`, over the pure `scrollbarThumb` in `src/components/scrollbar.ts`) for the scroll indicator, rather than copying any of them; the search pane's follow stays action-driven (`setSearchSelection`) by design, and DiffView's variable-height windowing stays bespoke. A windowed list mounts only its visible slice, so OpenTUI's native `<scrollbox>` thumb has no real content to measure (#187 lost it that way): `ListScrollbar` hand-draws a reserved width-1 column whose thumb is derived from the same `(rowCount, viewport, scrollTop)` the window already tracks, one `<text>` (thumb cells `▐` in `theme.colors.scrollbar.thumb`, track cells transparent), painting nothing but staying reserved when the content fits so growth never shifts layout. DiffView keeps a real scrollbox (spacers give it measurable height) and deliberately hides its own scrollbar, so it does not use `ListScrollbar`. The references overlay (`src/components/ReferencesOverlay.tsx`) is a palette-family overlay that windows this way too, not with a native `<scrollbox>` + `scrollChildIntoView`: that imperative scroll-follow was unreliable in the full app (its computed delta depends on render/layout timing under `viewportCulling`, so the selection moved but the viewport never followed), while the windowed slice derived from `referencesScrollTop` (followed off the cursor's row via `followListWindow`, mapping the results-space `referencesIndex` to its row index since per-file headers offset it) is declarative and matches the panes. Its flattening is the pure `buildReferenceRows` in `src/intel/references.ts`.
- Sidebar width resizing writes a single overridable signal, `sidebarWidthOverride`: keyboard `[`/`]` nudge it, `\` resets it to `null` (responsive default). The override is stored raw and clamped only inside the `sidebarWidth` memo, so a manual width never overflows a shrunken terminal yet is restored when the terminal grows back. Shrinking past the minimum collapses the sidebar (like an IDE pane) rather than clamping, moving focus off the tree the way the `ctrl-b` toggle does.
- Every subprocess runs through the `Process` service (`src/process.ts`, the one cross-cutting infra module that sits at the root rather than in a domain folder), an interruptible `Effect.acquireUseRelease` over `Bun.spawn` whose release kills the child; the git, file, and clipboard services and the startup load all compose it. There is no synchronous `Bun.spawnSync` path. Retry only transient git failures (the `index.lock` class), bounded, via `Effect.retry`; do not blanket-retry, a bad ref should fail fast. **Exception:** `src/editor/service.ts` (`EditorLive`) spawns directly via `Bun.spawn` without the `Process` service. `Process` pipes stdio and kills children on fiber interruption; for a terminal editor that needs inherited stdio and for a GUI IDE that should outlive the fiber, both behaviors are wrong. `openTerminal` inherits stdio and awaits `proc.exited`, suspending the renderer around it; `openIde` ignores stdio and awaits exit to surface a non-zero code. `openExternal` (the OS default app, `open`/`xdg-open` via the pure `openExternalCommand`) shares `openIde`'s detached fork-and-forget spawn, so it needs no renderer and lives as the plain `state.openExternally` action (`O`, or the command menu's `openExternal`), reached directly like every other data action. The same exception covers `src/upgrade/run.ts`: `stet upgrade` is a pre-TUI one-shot path (like `--help`/`--version`, dispatched by `parseCommand` in `src/cli.ts` before the renderer or any runtime is built) that classifies the install method from `process.execPath` and spawns the right channel's update (`install.sh` re-run, `npm`, or `brew`) with inherited stdio so the user sees its progress live, so it uses `Bun.spawn` directly and needs no Effect/`AppLayer`. No other domain may bypass `Process`. Version awareness lives in the same domain as a plain (non-Effect) module, `src/upgrade/release.ts`: GitHub Releases is the canonical source all channels derive from (`tag_name`, stripping the release-please `stet-v` component prefix down to the bare version), read with a plain `fetch` (a `User-Agent` header is required or GitHub returns 403) bounded by `AbortSignal.timeout` and swallowing every failure to `undefined`. There is **no cache, no env opt-out, and deliberately no `service.ts`** here, the same AppLayer-free rationale as `run.ts`: no AppLayer dependency, no `Process` child, no fiber-interruption resource to model. `upgrade` short-circuits with `stet X is already up to date` when `fetchLatestVersion` resolves a version that is not newer, and falls through to the channel update when the check fails. `state.checkForUpdate` runs the same check in the background on launch (non-blocking, independent of the git load) and stores a hit in `availableUpdate`.

## Code design

- **Extraction is a design decision, not a refactor.** When you see duplication, surface it and propose. Do not extract shared helpers, types, or components unprompted, regardless of usage count. Whether two similar blocks are one concept or two is a judgment you don't have the context to make alone.
- **Inline by default; extract only when it earns a name.** Don't pull out single-use helpers preemptively: inline at the call site unless the helper is reused, hides a genuinely complex boundary, or has a clear independent name that improves the caller. When a function grows several branches, let the main function read as the happy path and move supporting detail into small named helpers (`requireConfig`, `readMetadata`) kept just below it.
- **No speculative generality.** Don't add config options, generic parameters, or extension points for use cases that don't exist yet. The right abstraction for a future need is almost never the one you'd guess before seeing it.
- **Duplication is cheaper than the wrong abstraction.** Code that looks similar isn't necessarily the same thing. Deduping two unrelated pieces couples their futures; when one needs to change and the other doesn't, the abstraction has to grow conditionals or split back apart, both of which are worse than having kept them separate.

## Testing

- **Test behavior, not implementation.** Assert what a caller or user observes, not how the code achieves it. "Caller" and "user" scale with the unit under test: for a component it's the person clicking, for a function it's the code calling it. Tests that assert internals break on every refactor and prove nothing about whether the code works.
- **Prefer clarity over DRY in tests.** Inline setup, repeat literals, skip shared fixtures when they'd obscure the case under test. A test's job is to be readable in isolation; the pull toward DRY that makes production code better usually makes tests worse.
- **Test real behavior, not hypothetical behavior.** Cover the cases the contract actually promises. Do not manufacture edge cases the code doesn't claim to handle just to pad coverage.
- **Avoid mocks.** Test the actual implementation; reach for a mock only when a real dependency is genuinely unavailable, and never duplicate the code's own logic into the test.

## Writing

- **No em dashes.** Use a comma, period, or restructure the sentence instead.
- **No horizontal rules as section dividers.** Don't use `---` to separate sections when a heading is already doing that job. Exception: `---` is fine as a thematic break when there is no heading on either side.
- **No curly/smart quotes in prose.** Use straight quotes (`"`) not curly quotes. Applies to prose only; code blocks are verbatim and exempt.
- **Sentence case subheadings.** Write `## Like this` not `## Like This`. Exception: proper nouns and acronyms follow their standard casing.

## Docs

- **After introducing a new pattern, feature, convention, or structural change, ask whether `AGENTS.md` and/or `README.md` should be updated, then apply the changes.** Docs rot the moment the code moves without them. Catching the update at the point of change is the only time it reliably happens.
- **New or reworked UI surfaces clear the bar in `TASTE.md`:** design every state (default/hover/focus/active/disabled/loading/error), empty states included; theme tokens only; no layout shift; keyboard-operable; severity paired with its glyph; plain sentence-case copy.

## Implementing a change

- **Always work in a git worktree**, one branch per change; never commit directly on the default branch. Worktrees live under `.claude/worktrees/`.
- **Commit with `bunx gitzy`** (the project's conventional-commit prompt) rather than a raw `git commit -m`. It produces the `type(scope): summary` format with a leading gitmoji.
- **Keep the emoji**, and never add a co-author or AI sign-off trailer to commits.
- **PR description uses two sections and nothing else:**

  ```markdown
  ## What

  <what changed, in plain terms>

  ## Why

  <the problem or need it addresses>
  ```

  No trailing "generated by" / AI-attribution line, in the PR body or anywhere else.

## Deciding what to do

Judge every piece of work by whether it **should** be done (is it correct, is the current state wrong or inconsistent, does it serve the goal), and **never by ROI, cost, effort, or "is it worth it."** Do not label a known-wrong thing "low-value," "marginal," "an edge case," or "not worth it" to justify leaving it unfixed; reasoning by ROI is exactly what keeps work mediocre. ("The reference / competitor also gets it wrong" is a _gap_ argument, not a correctness one; it never makes a wrong thing acceptable.)

The only valid reason to stop short of doing the right thing is that it **provably cannot** be done: a demonstrated limit of the model or tools, not an assumed or cost-based one. "Hard," "heavy," "expensive," or "a lot of work" is never a reason to stop; "proven impossible / blocked" is. When unsure which it is, find out (try it, measure it, prove it) before deciding, and never declare a limit you have not proven.

Present choices by correctness and feasibility (real impossibilities, real _capability_ tradeoffs like portability or expressiveness), not by ROI.

## Fixing bugs

Assume a correct architecture has no bugs, so every bug is evidence that the architecture _permits_ it to exist, not merely that one code path is wrong. **Before fixing any bug, first diagnose the root cause: ask why the architecture allowed this bug to exist at all,** and whether it is one instance of a whole _class_ of bugs the same structure would keep producing.

Prefer fixes that remove the structural condition that let the bug exist, so this bug and others like it can no longer occur, over patches at the symptom layer (a guard, a special case, or a workaround that leaves the enabling structure in place). Reach for a symptom-layer patch only when the root-cause fix is **provably** infeasible or genuinely belongs in a separate change, never merely because it is larger or harder; when you do, say so and name the root cause you are deferring.

This is a thinking-first rule, not a mandate to refactor on every fix: the root-cause analysis is **always** required; reshaping the architecture to act on it is frequent but conditional on its being the right and feasible move.

## Pausing

- **Between phases:** after completing a discrete chunk of work, stop. Post a short summary, then ask before starting the next.
- **On uncertainty:** if a decision is not covered by existing rules or context, do not invent. Stop and ask.
- **On a debug loop:** if the same error persists after 3 consecutive fix attempts, stop. Report the error, what was tried, and the likely cause. Do not attempt a fourth fix without input.

## Scope guardrails (v1)

Do not implement a web preview, PR workflow, accept/reject protocol, agent integration, or a database. The tool must work in any git repo, not only this one or agent-created worktrees. Diagnostics are LSP-backed (read-only: `publishDiagnostics` is exactly what an IDE shows you), so a diagnostics-only LSP client is in scope; a full editing LSP client is not.
