# stet: implementation spec

The README is the source of truth for what stet does, its keys, and its non-goals. This file holds the invariants behind those features, the contract the README doesn't state. Read it before changing tree construction, the viewer, recency, diagnostics, scopes, or worktree handling.

## Architecture invariant

Git output is the synchronous source of truth. The git-backed file tree renders first; diagnostics arrive later as independent async decorations over the stable tree, so the basic view stays useful while checks run.

## Tree construction

- Source the tree from `git ls-files --stage` (tracked, carrying the file mode) plus `git ls-files --others --exclude-standard` (untracked, gitignore respected), union'd with the changed set so staged deletions stay visible.
- A symlink is identified by git mode `120000` for tracked files and by `lstat` for untracked ones; it renders with a distinct symlink icon, and its content/diff is the link's target path text (matching git), never the dereferenced target. The local-file read path uses `lstat`/`readlink`, so a link to a directory, a binary, or a missing target still reads as its one-line path.
- Ordering is directories-first, then alphabetical, always: stable under polling by construction, so the list never reorders under the cursor.
- Flatten single-child directory chains into one row.
- Tag each changed file with its stage state (staged, unstaged, mixed, untracked) from `git status`.
- Include untracked files in the changed set (except in the `staged` scope) and render them as all-added diffs.
- Go-to-file (`ctrl-p`) searches the same file universe as the tree.

## Scopes

`stet [ref]` defaults to `all` (worktree vs `HEAD`). `--staged` / `--unstaged` set the initial scope. `s` opens a scope picker (a static-list overlay like the worktree picker: `j`/`k` navigate, `enter` selects, `esc`/`s` close, opening on the active scope) over five scopes, in order: `all` (worktree vs `<ref>`), `staged` (`git diff --cached <ref>`), `unstaged` (plain `git diff`, ignores the ref), `session`, `last-commit`. The picker's explicit labels are what make `unstaged`'s ref-drop legible.

`session` ("since session start") diffs the worktree against the SHA `HEAD` pointed at when stet launched, captured once at startup and re-pinned on a worktree switch. Its base stays fixed as the agent commits, so it spans intervening commits plus uncommitted and untracked work: the full delta since you started watching. `last-commit` ("last commit") is the newest commit's own diff, `git diff <parent> HEAD`, with untracked excluded; on a root commit (no parent) it falls back to the empty-tree SHA so the whole first commit renders as all-added. Its right side is the literal `HEAD`, so it always follows the newest commit, and its parent re-resolves as `HEAD` moves so a new commit advances the window and re-keys the checks; a scope switch never registers as recency activity.

The picker is two levels: `enter`/`right` on its trailing `commits →` row drills into the last `LIMIT` commits (`git log`, newest-first, loaded on open; `esc`/`left` backs out), and selecting one applies a sixth scope, `commit`, that shows that commit as its own diff. A `commit` scope is `git diff <parent> <sha>` (the commit's first parent captured from `%P`, or the empty tree for a root commit), reusing the `last-commit` range machinery: it is a committed-tree-to-committed-tree range, so untracked files are excluded, and it is not a top-level picker row (never applied except through the drill-down). It is pinned by SHA, so it does not follow `HEAD`; a worktree switch resets it to `all` since the SHA has no meaning there. The header names the active commit by its subject; the commit list is a snapshot, so a commit made after opening the drill-down appears on the next open.

## Worktrees

`w` switches the active worktree in place and re-points the tree, diffs, refresh (watcher and poll), and checks at it, with no restart. The switch reseeds diagnostics from the new worktree's changed set, so the prior worktree's findings (including cross-file ones the server reported outside the changed set) never carry across. The switch loads only the changed set up front (the same shape startup seeds), so it re-points the instant the cheap diff commands resolve rather than blocking on a full `git ls-files --stage` over the worktree; the full tracked-file tree fills in on the repoFiles refresh (triggered on the re-key, then whenever the changed path-set shifts, plus a slow floor), exactly as at startup. The picker lists worktrees from `git worktree list --porcelain -z` (bare entries excluded) and marks locked and prunable ones.

Each worktree also carries a **summary** (`src/git/worktree.ts`, pure): `lastActivityAt`, when an agent or git last touched it. This is how stet answers "which worktree is an agent working in", and it stays inside the read-only, no-agent-integration guardrail: it never looks for an agent, only at what one leaves behind, so it works in any git repo.

**Activity is the most recent of three signals, and needs all three.** The obvious one, the newest mtime among the worktree's uncommitted files (`git status --porcelain=v1 -uall -z`, one call per worktree; `-uall` so an untracked directory is not collapsed to a `dir/` whose own mtime would not move when a file deep inside it is written), is on its own **actively wrong**: it goes to zero the moment an agent commits, which is exactly when the worktree was busiest, so a just-finished agent's worktree reads as dead. So it is maxed with the **`logs/HEAD` reflog mtime** (a commit, checkout, reset, rebase, or the worktree's own creation), a free `statSync` whose git dir is resolved without git (`<worktree>/.git` is a directory in the main worktree and a file naming the real dir in a linked one), and never rewritten by a `git status`, unlike `index`, so stet's own polling cannot fake activity into it. **HEAD's commit time** is the portable floor beneath the reflog for a repository with `core.logAllRefUpdates` off, fetched for every worktree in **one** subprocess (`git show -s --format='%H %ct' <sha>…`, using the HEAD shas `parseWorktreeList` already parsed, since all worktrees share the one object database). Every value is a real timestamp, **never the moment we noticed**, so a poll that runs late still reports an exact age.

Summaries are polled, not watched (only the active worktree earns an fs watcher), every `PEER_SUMMARY_MS` on a background fiber keyed on `repoRoot` like the refresh loop. The `git worktree list` on each tick is also the discovery mechanism, so it cannot be resolved once at startup: an agent can create a worktree mid-session. It is a cheap directory read, and a repository with a single worktree stops there, never paying for a `git status` it has no peer to compare against. `mergeWorktreeSummaries` keeps the previous map when nothing moved, so an idle repository's poll wakes no renderable.

A worktree stays active for **`WORKTREE_ACTIVE_MS`** (5 minutes), deliberately far longer than the 30s `RECENT_MS` a changed _file_ stays fresh for: that window is tuned to flash a row in the tree, whereas an agent pauses for minutes at a time (thinking, running tests, waiting on a build) and is plainly still working the whole time. Because the two windows differ, `recencyFraction` takes an optional `window` (defaulting to `RECENT_MS`, so the tree, file picker, and status bar are untouched); without it a worktree an agent is plainly still working in would grey out after 30s.

The **picker** is the one surface that reads the summaries, and it answers the whole question (_which_ worktree, not merely how many). Its row is the branch, its `locked`/`prunable` badges, and its age (`relativeTime`: `now`, `5m`, `3h`), tinted on the recency ramp over `WORKTREE_ACTIVE_MS` (`recency.fresh` fading into `recency.aged`, which is where `text.faint` sits) so a worktree an agent is working in glows and cools as it goes quiet. The age carries the recency rather than a `RecencyDot`, because `●` already means "current worktree" in this row and one glyph may not carry two meanings; the word is the signal, so the row reads under `NO_COLOR`. There is deliberately **no changed-file count** (a column repeating `clean` answers a question nobody asked) and **no per-row path**: the worktrees of a repository share a long common prefix, so a path left-truncated to a fixed column slices that prefix at a different offset on every row (`…trees/`, `…ude/`, `…ees/`) and cuts the leaf, the only part that differs, while no in-row path can fit anyway. The full path of the **highlighted** worktree instead gets its own fixed-height row above the instruction hint (the `ReferencesOverlay`'s detail-row shape), where a real path fits the overlay's width complete and untruncated. Rows are ordered by `orderWorktrees`, purely by how recently each was touched, with **no pin for the main worktree** (it is only the busiest worktree when it actually is). That order is computed **once, when the picker opens**, never on a summary refresh, so the list can never reorder under the cursor while it is open; the cells themselves stay live.

Every picker cell is a fixed-width `<box>`, never a padded string, and the width budget subtracts **four** cells from `overlayWidth`: two for the overlay's border and two for each row's own padding. Both rules are load-bearing and each has already caused a bug. A whitespace-only `<text>` measures zero cells in a flex row, so a padded empty cell slides every column right of it out of line; and budgeting against the padding alone overruns the row by exactly the border, which the renderer silently clips off the right edge (it ate the last character of every path).

The **header carries no activity cue**, and should not grow one back. It briefly shipped an `N worktrees active` line, which crowded the densest row in the app: joined to the identity by the same `·` that separates repo from branch, it read as a third peer token beside them, and it answered only the weaker half of the question (how many, never which). The picker answers the whole of it, so the header stays git identity.

If the active worktree is deleted out from under stet mid-session, the always-on refresh heartbeat is the detector: a git failure where the worktree root, or the repository's main worktree, no longer exists flags the deletion (every subprocess fails fast with a clear "working directory no longer exists" message rather than a raw `posix_spawn` error, because `Process` guards `cwd` existence before spawning). stet then auto-switches to the main worktree. The main worktree is resolved once at startup from the common dir (`git rev-parse --git-common-dir`, the `<main>/.git` whose parent is the main worktree) and lives outside any linked worktree's tree, so it survives that worktree's deletion without needing to be re-queried. When the main worktree itself is gone, the repository is gone (every linked worktree's git points into the deleted `<main>/.git`), so stet exits cleanly with a one-line message instead of looping on the dead directory.

## Overlays

Every overlay (the go-to-file `FileCombobox`, the `ScopeMenu` and `WorktreeCombobox`, the `HelpDialog` keybindings reference, the `ThemeCombobox`, and the `ReferencesOverlay`) reads as one command-palette family: a centered bordered box at the shared `overlayLeft`/`overlayWidth` and `zIndex 100`, a list, and a muted instruction-hint footer at the bottom (`↑↓ navigate · ⏎ <verb> · esc close`, tailored per overlay and naming only the keys that overlay actually honors). Input-driven overlays (`FileCombobox`, `ThemeCombobox`, `WorktreeCombobox`) are title-less, the filter placeholder is their label; the fixed `ScopeMenu` and the `HelpDialog` carry a short action-verb title (`switch scope`, `keyboard shortcuts`), and the query-less `ReferencesOverlay` titles itself with the request name (`references` or `definitions`). The `ReferencesOverlay` is the one member with no filter input, so like the `HelpDialog` its keys (nav, `enter` to jump, `esc` to close) live in the keymap rather than an input's `onSubmit`; it designs a loading, empty, and error screen in the list band and keeps a status-summary row (result and file counts) separate from the instruction hint. `esc` is the canonical close on all of them. The `HelpDialog` stays a read-only, dismiss-only reference (no filter input, so it is a dialog and not a combobox) and groups its keybindings under calm muted section labels (`navigation`, `viewer`, `tabs`, `workspace`, `layout`, `app`), spaced one row apart, rather than as one flat list; its entries come from the shared `KEY_HELP` registry (`src/help/keys.ts`) that `stet --help` also renders, so the overlay and the CLI help can't drift. The chrome is deliberately inlined per overlay for now (a shared overlay-shell component is a future extraction), so each overlay owns its own rows, swatches, and hover-preview.

Separate from that centered family is the **caret-anchored decoration** (`CaretCard`, the hover card today; peek/gutters later): a floating bordered box placed at the caret cell inside the viewer, not centered. The placement math is pure (`src/viewer/anchor.ts`: `caretCell` maps the caret to its on-screen cell, `placeCard` clamps the card to the viewer and flips it above the caret near the bottom edge), and the card mounts inside `DiffView` (a relative box wrapping the scrollbox) so it positions in viewer-content coordinates with no dependency on the header/status heights and never clips against the scrollbox. One content-agnostic `viewerDecoration` signal in `state` drives it (a status plus body lines, so a second consumer reuses the seam), and an orthogonal anchor closes the card on any caret move, scroll, file switch, or `esc` (`esc` claims the key before the find and global handlers).

## Theme switcher

`t` opens a theme switcher (an input-driven overlay like the go-to-file palette: a filter input, `↑`/`↓` or `ctrl-n`/`ctrl-p` navigate, `enter` applies, `esc` cancels), listing an `auto` entry (follow the terminal) followed by every registered theme (built-ins plus config themes). It is a runtime, session-only control: it writes nothing to disk, so a restart returns to the config selection. Navigating previews live, the highlighted theme becomes the active selection so the whole UI and the diff re-theme instantly through the one reactive seam (the same `selection` signal the config seeds and the appearance flip drives); the wheel and hovering a row drive the same highlight, so a pointer previews as it moves. The selection active when the picker opened is captured as the revert target: `esc` (or any cancel) restores it, `enter`/click commits the highlighted one. A single picked theme is a name, never a `{dark,light}` pair (that stays a config-only construct), so opening on a pair parks the highlight on `auto`.

## Live view

Refresh the tree, diff, and file content while the user watches. A debounced filesystem watcher (the worktree tree plus the resolved git dir, so staging in a linked worktree is seen) re-derives git state the instant a change lands; a slow safety poll is the floor that covers anything the watcher misses, so the worst case is poll-speed, never stale. The watcher ignores high-churn git internals an agent generates while it works (loose objects, reflogs, `*.lock` files, rebase/merge scratch, `COMMIT_EDITMSG`, `FETCH_HEAD`), which never change the rendered state; staging/commit/checkout still tick via HEAD, the index, and refs, and anything wrongly filtered is still caught by the poll floor. Preserve selection by path and the cursor across refreshes; reset the cursor only on a file switch.

## Viewer

- Unchanged files render full content read-only. `v` toggles a changed file between diff and full content.
- Full files render through the diff viewer as synthesized all-context patches.
- Per-file diffs are computed in-process, not via `git diff <ref> -- <path>`: `fileDiffSides` (`src/git/file-patch.ts`) resolves the scope's two endpoints (a `git show <ref>:<path>` / `:<path>` blob read on each committed/indexed side, a raw worktree read on the worktree side), and jsdiff builds the git-shaped unified patch the viewer already parses. In very large repos git's pathspec-limited diff-index walks the whole index (seconds per open), while a blob read stays O(path depth). The pathspec `git diff` invocation survives only as a fallback for the cases an in-process diff can't reproduce faithfully: a side that is missing/oversized, a jsdiff timeout, one-sided CRLF (git may eol-convert the worktree before diffing), or zero hunks against non-zero numstat counts. Binary files (flagged by numstat) skip the diff and instead fetch both sides' size and image dimensions (`git.binaryMeta`, reusing the same side reads) for the metadata surface below; untracked files keep the existing `git diff --no-index /dev/null <path>` path.
- The diff is rendered by stet's own windowed renderer (`src/diff/`, `src/components/diff/DiffView.tsx`) on `@pierre/diffs` (headless parse + Shiki highlight) and OpenTUI primitives, not the built-in `<diff>`. Invariants that keep it from wedging OpenTUI's scheduler: the line-number gutter is padded to a file-wide fixed width (never sized from the visible slice, so it never oscillates); only the visible row slice plus a small overscan is mounted, with top/bottom spacer boxes standing in for the off-screen rows so total scroll height is exact; in non-wrap mode each row is pinned to a single terminal row, so the spacer and `maxScrollY` math stays exact even when a grapheme (e.g. a variation-selector emoji) would otherwise lay a horizontally-clipped line out two rows tall and strand the file's last line; and each selection commits in two structure-identical phases (instant plain rows, then a Shiki-highlighted row upgrade) sharing the same row count and gutter width. Vertical scroll is the scrollbox's native scroll mirrored into the window; long lines scroll horizontally via a shared `scrollX` offset that shifts every visible line together while the gutter stays fixed (wrap mode reflows instead). Syntax covers any language Shiki bundles: a common set is warmed into the shared highlighter at startup, and a file whose language is outside that set has its grammar attached on demand (inferred from the file name) before its highlight phase, so a never-bundled language falls back to plain text without blocking the render.
- Binary files render a metadata surface, never raw bytes: `BinaryPreview` (`src/components/BinaryPreview.tsx`) centers the file type, image dimensions, and human-readable size (`imageMeta` reads dimensions from the header bytes of PNG/JPEG/GIF/WebP/BMP, no decode; SVG stays text), in file-content mode from the single viewed side and in diff mode as an old -> new delta, with an "open externally" affordance. It is the metadata an IDE shows for an image stet cannot draw as pixels (OpenTUI has no SIXEL/Kitty path, anomalyco/opentui#92), and it reserves the full viewer height so opening a binary never shifts layout. `O` (or the context menu / the `openExternal` command action) opens the file in the OS default app. Missing and oversized files render a short placeholder line. A file whose diff or content exceeds the render cap loads partially; a reserved footer row at the bottom of the viewer names the hidden-line count (`⋯ N more lines · f to load`) and `f` (or a click on the row) loads the rest. The footer is a standing property of the current file, so it lives at the content, not in the transient status bar; the row is reserved out of `viewerHeight`, so the diff never shifts under it.
- `/` opens an in-buffer find over the rendered lines: smart-case substring match (case-insensitive unless the query has an uppercase char), `n`/`N` cycle, `esc` clears, and a file switch ends it. Matches are highlighted at the line level; substring-range highlighting within a line is out of scope for v1.
- **Collapsed regions** fold or expand runs of lines behind a single non-navigable marker, all through one pure transform (`applyCollapsedRegions`, `src/diff/regions.ts`) over the raw row stream. Two kinds with opposite defaults, two `Set`s of keys: a **user fold** starts expanded and `z` at the caret collapses it behind `▸ N lines folded`. The region model is language-aware (`foldRegionsFor` dispatches on the file's `languageForPath`): code folds by indentation (`computeFoldRegions`, a line whose following lines are more indented heads a region), while markdown folds by heading section (`computeMarkdownFoldRegions`, an ATX heading down to the next same-or-higher heading, fence-aware). The other kind, a **git gap** (the elided-context `separator`), starts collapsed as `⋯ N unmodified lines` and `z` (or a click) expands it, revealing the unchanged lines sliced from the file's source (loaded lazily once per file via the git service's `fileSource`, with loading/error states). `z` unfolds a folded header, else folds the region enclosing the caret, else toggles the gap nearest the caret; clicking any marker toggles it. **Invariant:** the caret indexes only visible lines. The transform re-densifies each surviving line row's `navIndex` and rebuilds `navigable`, and every toggle remaps the caret to its file line (or the fold header when that line is now hidden), so the caret can never land inside a fold. Folds/gaps are per file and reset on file switch; a jump whose target line is folded clears the covering fold so it can still land (`revealLineForJump`). Wrap moved from `z` to `x`.
- The viewer carries a **word-granular caret** on the cursor line: `h`/`l` (and `←`/`→`) hop it identifier to identifier (`src/diff/words.ts`), wrapping across line boundaries (past the last word to the next line's first, past the first to the previous line's last), and the word it owns gets a cool neutral background highlight (`markRange` in `src/diff/spans.ts`, the `caret.wordBg` token); `Tab` returns focus to the tree (`h` no longer does). The caret is stored as a **precise UTF-16 offset** (`cursorColumn`), so a diagnostic jump lands on its exact column and copy emits `path:line:col`, but motion and the highlight are word-wise. It homes to the line's first word on every vertical move (`setCursorRow`) and a content reload; a jump snaps it to the word owning the target column. A click on the line-number gutter instead selects the line with **no symbol** (`caretLineLevel`): nothing highlights and `y` copies `path:line` rather than `path:line:col`; the next caret motion or content click re-selects a symbol. Two derived signals split the concerns: `caretWord` ("is the caret on a symbol, and which") drives the highlight and the code-intel requests, while `caretColumn` (gated only by `caretLineLevel`, not by landing on a word) drives the `:col` in copy/stats — so a diagnostic jump that lands in a gap still reports its exact column. Horizontal follow keeps the caret word in view (`followScrollX`); the word highlight follows the word through wrap, but horizontal follow is scroll-mode only.

## Navigation and tabs

The viewer is a set of tabs, each an ordered history of Locations (a Location is what restores the view: path, cursor file-line, caret column, scroll, and the `v`/`f` toggles). The pure model lives in `src/viewer/navigation.ts`; `state` keeps the live signals (`selectedPath`/`cursorIndex`/scroll) as the source of truth and the history as an orthogonal recorder that captures a Location on leave and applies one on arrive.

- **One preview tab.** At most one tab is the ephemeral _preview_ (rendered italic, the standard ephemeral-tab cue, which frees the color channel so a previewed changed file still shows its diff tint). All navigation — tree browse, jumps, palette, search, problems, `.` recency — targets the preview tab and replaces it in place: continuous tree browsing coalesces into one history entry, a deliberate jump pushes (truncating forward entries, browser semantics). Browsing never spawns or duplicates tabs; this is the calm single-buffer feel by default. The strip earns the viewer's title row only once a tab is pinned; a lone ephemeral preview shows the path instead.
- **Pin/unpin (`ctrl-t`).** Toggles the active tab. Pinning the preview makes it persistent and the next navigation opens a fresh preview; unpinning a pinned tab reverts it to the (sole) preview, discarding any stale preview so exactly one exists. Navigating to a file already open in a _pinned_ tab focuses that tab rather than previewing it, so a file is never in both the preview and a pinned tab. Double-clicking a tab in the strip or a file in the tree pins it too (pin-only, never unpins, unlike `ctrl-t`'s toggle); OpenTUI has no double-click event, so it is detected from successive mouse-downs.
- **History and tabs (`<`/`>`, `{`/`}`, `ctrl-w`).** `<`/`>` move back/forward within the _active_ tab's own history (no truncation), restoring that entry's exact cursor and scroll. `{`/`}` cycle tabs with wrap; `ctrl-w` closes the active tab — closing the sole remaining tab reverts it to the preview (a view always exists, so the floor is the ephemeral preview).
- **Remembered position.** A global per-path MRU viewport (cursor file-line + caret column + scroll), shared across tabs, is written on every navigate-away. A _fresh_ visit to a path restores its MRU position; back/forward restore the exact spot recorded on the history entry instead. Cursor is a file line (stable across refreshes); scroll is best-effort.
- **Coherence.** Scope stays a global lens, never captured per tab or per Location. Refresh and scope changes never reset the history (only a `navigate` resets). A worktree switch resets navigation to a single fresh tab seeded with the new worktree's selected file. The tab strip is chrome (non-selectable); tab hierarchy is typographic, not chrome: no tab backgrounds and no header band, the active tab is the one full-strength label (the others recede to muted), the active shows its full path while inactive tabs show the basename, and the preview tab is italic. Diff status is not on the strip; it lives in the tree and the viewer stats.

## Content search

- `ctrl-f` opens the project search pane, the first **main-area view mode**: a `mainView` union signal (`"file" | "search"`) swaps the Viewer's interior, so exactly one view is ever active and future panes extend the union rather than adding booleans. The pane is persistent: closing it (`esc`, or any navigation to a file, including a result jump) keeps the query, results, selection, and collapsed groups, so `ctrl-f` after a jump restores the result set instead of clearing it. A worktree switch clears the snapshot (stale paths) but keeps the query.
- The engine is `git grep` (`-I` to skip binary, `--untracked` so it covers the tree's universe, `--column` for the match column). It searches working-tree file content, not diffs. Literal `-F` with smart-case by default; a regex toggle (`ctrl-r`) swaps `-F` for extended regex `-E`, and a case toggle (`ctrl-x`) forces case sensitivity over smart-case. `--column` reports a byte offset; it is converted to UTF-16 units so a jump lands the caret on the exact column even on non-ASCII lines.
- Scope defaults to the changed set (pathspec-limited to `git status`'s changed files, so it follows the active all/staged/unstaged scope); `ctrl-g` toggles between the changed set and the whole repo, and `ctrl-s` opens the scope picker without leaving the pane; the scope cell names the effective universe (`repo`, or the active diff scope: `changes`, `staged`, `unstaged`, `session`, `last commit`). A filter field narrows by pathspecs (space-separated, git wildmatch): plain tokens include, a `!` prefix excludes (mapped to git's `:(exclude)`, so includes union and excludes subtract; an exclude-only filter matches everything else, and raw pathspec magic passes verbatim). Under the changed scope the filter drives the grep and the changed set filters the results after, since include pathspecs union rather than intersect.
- Results group by file with per-file headers (match count, collapsible with `enter`/`h`/`l`), two lines of context around each match (overlapping context ranges merge into one excerpt, `⋯` separates non-contiguous excerpts), and Shiki syntax highlighting per file language through the diff's shared highlighter (highlighted on demand for the visible window; rows render plain and upgrade in place). Context comes from bounded file reads, not `git grep -C` (whose `-z` output makes match and context records indistinguishable); an unreadable or oversized file degrades to match-only rows. `enter` routes through the same `jumpTarget` path as problems navigation, landing on the line and column and escalating to full-file view when the line is outside the diff.
- Keyboard while the pane is focused: `tab` cycles query → glob → results, `↓` enters the results, `j`/`k` walk navigable rows (matches and headers; context and gaps are skipped), `ctrl-d`/`ctrl-u` half-page by _viewport_ rows (snapping to the nearest navigable row). Only the query/glob inputs swallow printable keys, and they keep their readline chords (`ctrl-a`/`ctrl-e` line home/end, `ctrl-w`/`ctrl-u`/`ctrl-k` kill); `ctrl-p` (palette) and the deliberately stolen `ctrl-b` (sidebar, with preventDefault) pass through even there. In results focus unhandled keys fall through, so the global bindings (`q`, `?`, `ctrl-p`, `p`...) keep working, matching the problems pane. A single click on a result selects it (a focus-intent click never navigates the view away); a double click, like `enter`, opens it. `esc` closes panels innermost-first: the problems panel, then the search view, then the app. File-view keys (`v`, `f`, `z` fold, `x` wrap, `ctrl-t`/`ctrl-w` tab pin/close, `F12`/`Shift+F12`/`K` intel, `y`/`Y` viewer copy) are inert while the search view shows, since they would act on a viewer that isn't on screen; in the results, `e`/`o` open the selected result in the editor, `y` copies its reference (`path:line:col` on a match), and `g`/`G` jump to the first/last result. Any navigation to a file (a result jump, the palette, problems, recency, history) reveals the file view, including a jump that lands on the already-selected file. The chrome rows (query, filter, summary, footer) are fixed-height in every state, so idle/searching/no-matches/error swaps never shift layout.
- The query is debounced and runs through the interruptible `Process` service, holding the previous snapshot until the new one resolves; a failed grep (a half-typed regex) keeps the prior results under an error notice. While the pane is open the search re-runs on every git-model refresh (and on reopen), so results, context lines, and jump targets track the working tree as an agent edits it. Results are capped to keep the pane bounded; hitting the cap is surfaced as a trailing `+` on the count.

## Recency

Recency markers come from an append-only in-memory activity event log (the seam for a future persistence layer). They decay silently: fresh under 5s, recent under 30s. `.` jumps to the latest activity. A scope switch is not activity. The status bar surfaces that latest activity, the most recent changed file, ahead of the leveled status, presented like a tree row: the path in its git change-kind color plus a recency dot, both fading toward the background across the window (no elapsed-time number), so it reads as a changed file and recedes silently as it ages.

## Provenance

`a` toggles a per-line provenance rail in the viewer, git blame reframed for reviewing agent output: the axis is not "who," which collapses to you-or-your-agent for anything recent, but "where on the file's history does this line sit, and how much of it is yours." Off by default. Each line, blamed from `git blame --porcelain` of the working-tree file, falls in one of five bands on a scrutiny timeline (oldest to newest, each tier more recent and more yours), classified against three anchors: the **file's first commit** (`git log --diff-filter=A`, per file), the **branch base** (`git merge-base HEAD <default-branch>`, where the default branch is `origin/HEAD` or a local `main`/`master`), and the **session base** (the SHA `HEAD` pointed at when stet launched, the same anchor the `session` scope uses).

The bands, in classification order so a newer tier wins an overlap: **uncommitted** (blame's all-zero sha), **this session** (commit in `sessionBase..HEAD`), **this branch** (commit in `branchBase..HEAD`; that set is a superset of the session set, so a line reads this branch only when its commit is in it but not in the session set, which is checked first), **initial** (the commit that introduced the file), or **changed** (any other committed line, the file revised on the base branch's history). The two commit sets are both `commitsSince(base)` (`base..HEAD`), so no range query is needed; each anchor degrades independently (no default branch folds the branch tier away, a file born on the branch has no initial tier).

The rail is a fixed one-cell column at the far left of the gutter, drawn only when on, so turning it on grows the gutter by exactly one column and never oscillates; a row with no working-tree line (a pure removal) and a row git cannot attribute stay blank. The band is a Block Element weight ramp (thick to thin: uncommitted `▋`, session `▌`, branch `▍`, changed `▎`, initial `▏`), the same block-not-box-drawing choice the change bar makes, for the same terminal-metric reason; the `changed` glyph shares the change bar's `▎` but never collides, since `changed` lands only on committed context lines where the change bar is blank. Color is one neutral brightness ramp over the same order (brightest at uncommitted fading to the faint neutral at initial, the five-member `provenance` token group), reinforcing the weight rather than adding a hue, since the rail is read-only inspection, not a warning or a status. The rail conveys the ordering; five weights are not all distinguishable in one cell under `NO_COLOR`, so the band reads as a gradient (thicker is more recent and more yours), not a per-cell label. An untracked or wholly-added file is unblameable, so it is not blamed at all; every line reads uncommitted from the model. While the rail is on, the caret line's commit fills the whole status bar as a blame inspector, led by its band glyph (the same mark the rail draws, the way the recency dot leads the recent-file path): `author · age · subject` for a committed line, or `uncommitted · working tree` for a working-tree line, taking the key-hint and recent-file cues' place. The band is the glyph, not repeated in words; the age carries the recency it orders. A transient message (an action notice, a cursor finding, an in-flight intel pull) preempts it, so an error or acknowledgment is never hidden behind blame. The blame reads, the file's first commit, and the two commit sets are fetched only while the rail is on, off the diff's refresh trigger, never on the diff hot path. It stays pure read-only inspection: no trust score, no "AI wrote this" claim (git cannot know), no judgment.

## Diagnostics

- Diagnostics come from language servers (LSP) over stdio, collapsed into one source; each diagnostic keeps its LSP `source` label (e.g. `typescript`, `oxc`, `biome`). The registry is data-driven and servers may overlap on a file type: `typescript-language-server` type-checks and `oxlint --lsp` lints the same JS/TS family, so a changed file runs through every server that claims its extension and the per-file results merge (union the findings; the strongest badge state wins, so a degraded server never overrides another's real result). A server may additionally be repo-gated by a `detect` predicate, running only where the repo opts into it: `biome lsp-proxy` activates only when a `biome.json`/`biome.jsonc` is present (mirroring an editor's Biome extension), so it neither downloads into repos that don't use it nor duplicates oxlint's JS/TS findings, and it covers `json`/`jsonc`/`css`/`graphql` beyond the JS/TS family. A repo-gated server that is off counts as no server for its files, so a Biome-only file (css/graphql) is `unavailable` in a non-Biome repo, never falsely clean. Two always-on validation servers cover the rest: `vscode-json-language-server` gives schema-aware `json`/`jsonc` validation in every repo (overlapping but complementary to Biome's json lint, which does no schema validation) and `yaml-language-server` covers `yaml`/`yml`; both push diagnostics only. Adding a disjoint language stays one language entry (its file types, matched exact-filename then extension, each mapped to its LSP languageId, plus its ordered server list) and one registry entry per new server. The same model is user-extensible from config (`languages` in `~/.config/stet/config.jsonc`): a config language declares extensions/exact filenames and servers (built-in names or inline commands, with `initializationOptions`/`settings` passed as data), resolved like themes (per-field merge over the built-in, every problem surfaced as a notice, one bad entry never sinking the rest) and registered at startup before the first run; a config language can never silently shadow another language's file types. Inline servers resolve repo-local then PATH, are never provisioned, and declare intel capabilities optimistically (the handshake-advertised set stays the authoritative gate). The trust boundary is the user's global config only: stet never reads server commands from the inspected repository, which would be arbitrary code execution on launch.
- Retrieval is hybrid: push and pull are concurrent channels, never either/or. Every server gets the push baseline: after a document reaches it the server pushes `textDocument/publishDiagnostics`, the run waits for freshly-sent documents (plus any held one still unpublished, so a late publish lands and counts) with a short capped settle, and the client must advertise `publishDiagnostics` and `synchronization` in `initialize` or servers stay silent. A server that advertises `diagnosticProvider` additionally takes the pull path for the run: one `textDocument/diagnostic` request per opened file (request/response, no settle heuristics), with the previous answer's `resultId` echoed back so an `unchanged` report reuses the cached items, `relatedDocuments` full reports landing on their own paths, and the push bucket still unioned per uri (a hybrid server like rust-analyzer answers pulls with native findings while pushing cargo-check ones). A pull that times out leaves the file `pending` (the next run retries); a pull the server rejects marks it `failed`, never falsely `clean`. The client advertises `textDocument.diagnostic` (with `relatedDocumentSupport`) and `workspace.diagnostics.refreshSupport`, and answers a server's `workspace/diagnostic/refresh` nudge by re-running checks for that repo. Workspace-wide pulls (`workspace/diagnostic`) are deferred until a built-in server provably advertises them (rust-analyzer, with the binary provisioning channel). A server that pulls its settings (oxlint) also needs `workspace.configuration` advertised and its `workspace/configuration` request answered; both derive from plain registry data (`initializationOptions`/`settings`, with `{repoRoot}`/`{repoUri}` substituted per repo in any string leaf), so a config layer can express them verbatim, and a `handshake` closure remains the escape hatch for shapes data cannot express (no built-in uses it). Per-server readiness hints stay deliberately unadded: the first-progress-end gate plus its backstop has no demonstrated failure. Of the current built-ins, oxlint, the json server, and ruff advertise pull and take that path; typescript, yaml, and basedpyright stay push-only.
- Discovery has three tiers, preferring the target repo: a repo-local binary (`node_modules/.bin`), then one on `PATH`, then a server stet downloads into its own cache (`~/.cache/stet/lsp/<language>/<version-key>`) if neither is present, never `bunx`. The download is a one-time background install, deduped per language, so diagnostics work out-of-the-box without a manual server install (the way Zed and opencode provision); opt out with `--no-lsp-download` or `STET_NO_LSP_DOWNLOAD`. Two channels, both pinned exactly in the registry: `npm` installs pinned package versions (this channel needs `npm` on the user's PATH), integrity-verified against npm's immutable published versions, never a bare name that resolves `@latest`; `binary` downloads a pinned GitHub release asset (a gzipped single binary, rust-analyzer's shape, or a `tar.gz` cargo-dist archive the binary is extracted out of by basename, ruff's shape) and verifies its sha256 against the whole downloaded asset before anything is written or marked executable, so a moved tag or tampered asset can never run. Servers with no upstream prebuilt binaries (gopls) stay repo-local/PATH-only; provisioning never builds from source. The cache dir is keyed by a digest of the exact pinned channel, so bumping a pin (e.g. to patch a vulnerable server) lands in a fresh dir and re-provisions instead of reusing the stale cached binary. The install is bounded by a timeout, so a fetch that cannot complete (offline, air-gapped, proxied, locked-down CI) degrades the file to `unavailable` rather than hanging in `pending` forever. Each server is pooled per `(language, repoRoot)` and held warm across runs by a **document keeper**: the changed set stays open on the connection (`didOpen` once, then a full-text `didChange` only when a file's on-disk content moved, nothing when it didn't), a file that leaves the set or vanishes from disk is closed, and each send commits with its bookkeeping as one uninterruptible step so an aborted run can never strand a document. Versions are per-uri monotonic for the connection's lifetime, owned by the transport. Every run reconciles the resident keepers: a worktree switch releases the previous repo's, a language whose changed set dropped to zero releases its own (closing their documents, so a pooled server re-held later starts clean), app teardown does the same via scoped finalizers, and a server that crashes mid-session is detected on the next run and rebuilt with its documents reopened fresh.
- Formatting is intentionally not a diagnostic: it is an action, not a finding, and stet only inspects.
- Retain findings for every reported path, not just changed files (a change can surface errors elsewhere).
- Surface in the problems panel (`p`), as a severity glyph (`✖`/`⚠`/`ℹ`, the strongest severity on the line) in the viewer's gutter, and as per-file markers in the tree. `n` jumps to the next file with findings. The gutter reads left to right as an optional provenance rail (only when `a` is on, see Provenance), the change bar, the line number, the severity glyph, then the code, carrying orthogonal signals on separate channels: the far-left change bar (a `▎` ¼-block, green for adds / red for removes, nothing on context) and the number digits take the diff state (add green / remove red / context neutral) while the glyph cell takes the diagnostic severity. The bar is a single glyph, add vs remove told apart by color alone. That is a deliberate call favoring equal footprint over the `NO_COLOR` add/remove distinction the `+`/`-` column used to carry; the diagnostic glyph stays glyph-paired regardless. The bar is a Block Element (reliably one cell wide); Box-Drawing glyphs are avoided because their rarer forms fall back to a mis-metriced font and misalign the gutter in some terminals. The glyph only ever lands on an added or context row, since a removed line has no new-line number to map findings by, and both the bar and glyph cells are reserved on every row so neither ever shifts the diff. A changed line also carries a faint full-row add/remove background tint (gutter and code share one color per line, edge to edge), kept dim so it reads as a block cue without desaturating the syntax highlighting; the gutter tint follows diff state only, so a find match stays a content-only highlight over it.
- Late diagnostics fill badges and markers in place and never reorder the tree.
- Badge states are explicit: `pending`, `clean`, `findings`, `failed`, `unavailable`. Missing or empty diagnostics never render as clean: a file the server has not published for yet (cold start) or whose server is still downloading stays `pending` (clean means the server published an empty set), a file whose language has no server or whose server cannot start (or download is disabled) is `unavailable`, and a file that changes returns its badges to `pending` until checks re-run.
- While any server is downloading, the status line shows a live `installing <lang> server…` (a count for several), sourced from the provisioner's start/finish, not the check run, so it appears promptly and clears the moment the download resolves. It outranks the ambient `checking…`/activity line but yields to the transient tiers above it (an in-flight intel pull, a held action notice, a cursor-on-finding message).
- Checks run at startup, on `r`, and automatically once the repo has been quiet for ~2s after activity. New-vs-baseline diagnostics are deferred.

## Code intelligence

- Beyond the push-diagnostics poll, stet makes on-demand read-only `textDocument/*` pull requests over the same warm `LanguageServers` pool (`src/intel/`). A pull is a `didOpen` → request → `didClose` bracket on the first acquired server that advertises the matching capability, parsed from the server's `initialize` reply into a capability set on the `ServerHandle` (so oxlint, which advertises none, drops out and `typescript-language-server` answers). Most pulls send one request; call hierarchy sends two (a `prepare` then a resolve) inside the one open document, sharing a single project-load wait and one close. Reads only, exactly like `publishDiagnostics`: no edit, format, or rename request is ever sent.
- A location reply carries absolute `uri`s; the service relativizes each to a repo path (reusing `relativize`, the same boundary the diagnostics checker uses). A target outside the repo (a definition in `node_modules`) stays absolute, and the caller skips it rather than opening an out-of-tree path. The pull bracket is generic over the reply shape: the location callers relativize inside their own `normalize`, while hover returns plain text and skips that step.
- A pull shares the pooled connection with the diagnostics flow, so its open/close bracket could race a concurrent open of the same document (a poll mid-settle, or a second pull). The transport refcounts each document per `uri` (`openDocument`/`closeDocument`): `didOpen` is sent only for the first holder and `didClose` only when the last holder releases, so a second open no longer resets the server's view of a doc another holder still needs, nor drops one out from under it. Both the intel bracket and the diagnostics open/settle/close go through it.
- A pull waits for the server's project to finish loading before it requests. The client advertises `window.workDoneProgress` in `initialize`, the transport tracks the project-load `$/progress` lifecycle, and `whenProjectLoaded` resolves on the first `end` (or on connection close). Until then a cold `typescript-language-server` answers `textDocument/definition` from the local import binding instead of resolving cross-file, so F12 on an import would stop at the import line; the gate is what makes a cold/slow server (a large monorepo) return the real source instead. The wait is interruptible (the caller aborts on the next navigation) and capped by a generous backstop, and it is a no-op once the project is warm.
- Go to definition (`F12`) resolves the symbol under the caret and jumps via the existing `jumpTarget` path (a cross-file jump pushes history, so `<` returns to the call site); a single in-repo target jumps, and more than one (an overloaded symbol) opens the references overlay labelled `definitions` rather than a blind jump to the first (picking a result with `enter` or a click jumps via `jumpTarget` and dismisses the overlay). A request failure degrades to a notice, never an exception into the UI. While the pull is in flight the status bar carries an in-flight indicator (`resolving definition…`, an `info`-level status), set on the keystroke and cleared when the pull settles (jump, notice, or a superseding request), so the wait for a cold server never reads as a dead keypress.
- Find references (`Shift+F12`) resolves every use of the symbol under the caret (`textDocument/references` with `includeDeclaration`) and opens the query-less references overlay: results grouped by file, each row a tabular `path:line:col` plus the trimmed source line; picking a result with `enter` or a click jumps via the same `jumpTarget` path and dismisses the overlay, while `esc` closes it without navigating. It guards exactly like definition, opens at once in a loading state, and resolves in place to the list, an empty screen (`no references`), or an error, so the wait never reads as a dead keypress without borrowing the status bar. Each result's source line is read locally for the preview (a missing or binary file still lists the row, without one); go-to-definition's multi-result case reuses the same overlay.
- Find implementations (`Shift+I`) resolves the concrete bodies of the symbol under the caret (`textDocument/implementation`) and behaves exactly like go-to-definition: the reply shape is identical (`Location | Location[] | LocationLink[]`), so it reuses `normalizeDefinition` and the same single-jump-vs-multi-overlay path. A single in-repo target jumps via `jumpTarget`; more than one opens the references overlay labelled `implementations` (a non-directional overlay, so `Tab` is a no-op). It guards exactly like definition, carries the same in-flight status indicator (`resolving implementations…`), and a failure degrades to a notice, never an exception. It is distinct from definition: definition lands on the abstract declaration, implementation jumps past the abstraction to the concrete bodies (an interface/abstract member lists every implementor; a plain concrete symbol collapses to a single jump). Client and server must both advertise the capability (`implementationProvider`, gated on the `implementation` client cap declared in `initialize`).
- Call hierarchy (`Shift+H`) lists the callers of the function or method under the caret in the same references overlay. It is a two-step pull: `textDocument/prepareCallHierarchy` returns an opaque item (its server-defined `data` preserved), then `callHierarchy/incomingCalls` (who calls this) or `callHierarchy/outgoingCalls` (what this calls) resolves the edges, both inside the one open document with one project-load wait, closing once. `Tab` flips the direction and re-resolves the same symbol (superseding the prior direction via the shared intel controller); the overlay label and footer say which way you're looking, and a plain references/definitions overlay (no direction) leaves `Tab` a no-op. It guards exactly like definition (caret on a symbol, on a line that exists in the file), opens at once in a loading state, resolves in place to the list / an empty screen / an error, and normalizes each caller or callee to its name range so a row jumps via the same `jumpTarget` path. Client and server must both advertise the capability; a server without it (`typescript-language-server` advertises `callHierarchy` but not type hierarchy) yields the empty state. Type hierarchy reuses this exact two-step machinery and is a deliberate follow-up, gated on a server that provides it.
- Hover (`K`) shows type + docs for the symbol under the caret in a caret-anchored card (`textDocument/hover`). The reply is parsed into ordered code/prose segments (the fence delimiters dropped): each code block is syntax-highlighted in its language through the same shared Shiki highlighter and active theme the diff uses (so the signature matches the diff's colors and re-themes with it), and prose renders plain. It guards exactly like definition (caret on a symbol, on a line that exists in the file) and degrades to an empty/error card, or to uncolored code when a language can't be highlighted, rather than throwing. Keyboard-only in v1; pointer-hover is a deliberate follow-up.
- Find symbols (`S`) lists the open file's symbols (`textDocument/documentSymbol`) in a query-less palette-family overlay, each row a kind glyph (a Nerd Font codicon gated on `iconsEnabled`, a text tag otherwise) plus the symbol name and its `line:col`, indented to mirror the file's structure. Unlike the caret pulls it needs only an open file (the whole document is the query), so it guards on `selectedPath` alone; picking a symbol with `enter` or a click jumps via the same `jumpTarget` path (always within the open file, escalating to full-file view when the line is outside the diff) and dismisses the overlay. It opens at once in a loading state and resolves in place to the list, an empty screen (`no symbols`), or an error. The client advertises `hierarchicalDocumentSymbolSupport` in `initialize`, so a server returns the nested `DocumentSymbol[]` (flattened pre-order into a depth-tagged, position-ordered outline); a server that still replies with a flat `SymbolInformation[]` degrades to a depth-0 list. Its own controller (not the shared definition/references one, since it owns a separate overlay), and it closes on repo or file drift.
- Non-empty pull replies are cached per `(repoRoot, path, line, character, method)` in bounded LRUs, so repeating a caret's hover/definition/references during a quiet period skips the LSP round-trip; an empty reply is never cached (it may be a transient warm-up miss). The cache keys off working-tree file content (what the server reads), so it is invalidated by the filesystem watcher's actual write events, not by the git diff model: any tracked working-tree write (an edit, a revert to baseline, or a deletion) drops the repo's cached replies, while a pure baseline move (a commit, staging, or a scope re-resolve, which touch `.git` not tracked files) leaves them intact. This is why the watcher surfaces the worktree-relative paths written per debounced batch and the consumer invalidates only when one is a file git tracks (or already counts as changed): the watcher also sees gitignored churn an agent generates (`node_modules/`, `dist/`), which must not wipe the warm cache, so raw watcher activity alone is not the trigger. The watcher is the primary trigger but is best-effort (a platform without inotify, a sandbox, a network filesystem, or a dropped event), so the safety poll is the fallback: whenever a refresh (watcher- or poll-driven) surfaces a model whose newest changed-file mtime advanced, the cache is invalidated too. That mtime signal is deliberately conservative — a commit, staging, or scope re-resolve moves no mtime — so the fallback never over-invalidates on a baseline move; it is redundant with the watcher on a healthy platform and the correctness floor where the watcher misses, keeping intel at worst poll-stale, never indefinitely stale. Both paths apply the same behavior: invalidation is repo-wide, not per-path, because a references or call-hierarchy result queried from one file can point into another; a worktree switch invalidates the switched-to repo, whose cache may hold entries that changed while its watcher was inactive; and an in-flight pull whose round-trip straddles any invalidate is not re-stored (a generation guard), so it can't strand a stale entry read from pre-edit content.

## Version awareness

- The latest version comes from GitHub Releases (`tag_name`, strip the `stet-` release-please component prefix, then a leading `v`, so `stet-v0.5.0` parses), the canonical source the install script, the brew tap, and npm all derive from. The lookup is a plain `fetch` (`src/upgrade/release.ts`) with a required `User-Agent` header, bounded by a timeout, and resolves to `undefined` on any failure. There is no cache: the check runs on every launch and fresh on every `upgrade`.
- The background check runs once on launch, non-blocking and independent of the git load. A newer release is held in `availableUpdate` and printed once on a clean keyboard quit (`q`/`esc`/ctrl-c), after the alt-screen is restored. It never prints on a crash or external kill (those backstops do not route through `quit`), and never interrupts the running session.
- `stet upgrade` compares the current version against the latest before spawning: when the latest is not newer it prints `stet X.Y.Z is already up to date` and exits without running the channel update; when the check fails (no network) it falls through to the channel update, leaving each tool to resolve `@latest` itself.
